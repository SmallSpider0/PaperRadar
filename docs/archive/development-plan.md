# PaperRadar 开发方案（3阶段版）

## 1. 项目定位

PaperRadar 是一个部署在本服务器上的内部科研工具，用来持续跟踪特定计算机研究方向的最新论文进展。首期聚焦安全方向四大顶会（USENIX Security、NDSS、IEEE S&P、ACM CCS），目标不是一次做全，而是先做出一个**能稳定抓、稳定存、稳定搜、稳定订阅**的最小可用系统。

本项目按 **3 个阶段**推进：

- **阶段一：核心闭环**
- **阶段二：阅读与报告增强**
- **阶段三：订阅完善与可视化扩展**

---

## 2. 总体原则

### 2.1 先做成，再做强

首期只做能真正跑起来、能长期维护的核心功能，不在一开始堆太多高级特性。

### 2.2 合规优先

- 默认只抓论文元数据与来源链接，不默认抓取 PDF 全文
- 当且仅当进入报告生成、人工点选解析或其他明确需要全文的流程时，才按需抓取 PDF
- IEEE S&P / ACM CCS 首期默认 `metadata-only`
- 不做反爬绕过，不碰高风险自动全文抓取

### 2.3 单机优先

以当前服务器直接开发与部署为目标，优先选择低复杂度、易运维方案。

---

## 3. 三阶段规划

## 阶段一：核心闭环（首期）

### 3.1 目标

完成一个最小可用版本，让系统具备：

- 论文抓取
- 元数据入库
- 按需全文抓取能力
- 基础检索
- 基础订阅匹配

一句话：**先把“新论文进来后能被存下、搜到、订阅到”打通。**

### 3.2 范围

#### 数据源
- USENIX Security：默认抓论文列表、详情页、摘要信息与来源链接
- NDSS：默认抓论文列表、详情页、摘要信息与来源链接
- IEEE S&P：只入库元数据
- ACM CCS：只入库元数据
- 可用 OpenAlex / Crossref / DBLP 做元数据补全

#### 存储
- PostgreSQL 存元数据、订阅、通知、按需解析结果
- 本地目录仅保存按需抓取下来的 PDF 原件与产物
- 本地目录存解析结果
- pgvector 存向量（仅对已按需解析的内容建立）

#### 检索
- 标题 / 摘要 / chunk 的关键词检索
- 基于研究方向描述的语义检索
- 支持按会议、年份过滤

#### 订阅
- 支持“研究方向描述订阅”
- 支持“会议订阅”
- 新论文入库后做匹配并生成通知记录

#### 页面
- 最小前端页面：
  - 搜索页
  - 论文列表页
  - 论文详情页
  - 订阅列表页

### 3.3 阶段一明确不做

- 不做 HTML 报告生成
- 不做 PPT 生成
- 不做 Nano Banana 配图链路
- 不做复杂图谱可视化
- 不做公网多用户权限系统
- 不做分布式向量数据库

### 3.4 阶段一交付物

- 可运行抓取器（USENIX / NDSS）
- 元数据增强接入（OpenAlex / DBLP / Crossref，至少 1 个）
- PostgreSQL 数据库 schema
- 按需 PDF 抓取与 hash 记录
- 按需解析入口与 chunk / embedding 入库能力
- pgvector 检索能力
- 基础搜索 API
- 基础订阅 API
- 最小前端可用页面
- systemd + nginx 部署模板

---

## 阶段二：阅读与报告增强

### 3.5 目标

在核心闭环稳定后，让系统从“能搜到论文”升级到“能快速消化论文”。

一句话：**把论文变成更容易阅读、汇报和复用的结构化知识。**

### 3.6 范围

- 接入更完整的 PDF 结构化解析（GROBID 主导）
- 生成 HTML 阅读页
- 生成单篇论文报告
- 生成主题性汇总报告
- 生成 PPT 初版
- 把 HTML / PPT 产物入库

### 3.7 阶段二交付物

- HTML 阅读模板
- 单篇论文总结页
- 主题报告生成器
- PPT 生成器
- 报告产物管理表
- 报告查看页面

---

## 阶段三：订阅完善与可视化扩展

### 3.8 目标

在核心链路和阅读链路成熟后，增强持续跟踪能力和信息呈现能力。

一句话：**让系统从“内部工具”成长为“稳定的研究跟踪台”。**

### 3.9 范围

- 更丰富的订阅模型
  - 关键词组合订阅
  - 作者订阅
  - 多条件过滤订阅
- 推送渠道扩展
  - Webhook
  - 邮件
  - 站内消息增强
- 可视化增强
  - 趋势图
  - 会议 / 年份统计
  - 作者 / 主题关系图
- 视情况接入 Nano Banana 生成图示

### 3.10 阶段三交付物

- 多种订阅规则
- 推送渠道扩展
- 统计与趋势面板
- 可视化模块
- 配图生成工作流（可选）

---

## 4. 推荐技术选型

### 后端
- Python
- FastAPI

### 异步任务
- 首期：简单 worker + cron / APScheduler
- 后续：Celery + Redis

### 数据库
- PostgreSQL
- pgvector
- PostgreSQL Full Text Search

### 前端
- Next.js

### 存储
- 首期：本地磁盘目录
- 后续：可切 S3 / MinIO

### 解析
- 首期：按需 PDF 文本抽取 + 可扩展 GROBID 接口
- 二期：GROBID 主流程

### 部署
- systemd
- nginx
- PostgreSQL / Redis 本机服务

---

## 5. 首期（阶段一）建议落地顺序

1. 建项目骨架与目录
2. 建数据库 schema
3. 完成 USENIX / NDSS 抓取器
4. 完成论文去重与元数据入库
5. 完成按需 PDF 抓取接口与本地存储
6. 完成按需 chunk 切分与 embedding
7. 完成搜索 API
8. 完成订阅 API
9. 完成最小前端页面
10. 完成本机部署模板

---

## 6. 成功标准

阶段一完成后，至少要满足：

- 能自动抓到 USENIX / NDSS 新论文元数据
- 能把论文元数据和来源链接稳定存下来
- 能在需要时按需抓取 PDF 并完成解析
- 能根据研究方向描述检索相关论文
- 能创建订阅并在新论文入库时命中
- 能在网页中查看论文基本信息与搜索结果

如果这 5 件事都稳定成立，PaperRadar 的第一阶段才算真正完成。

---

## 7. 一句话结论

**PaperRadar 先做成“论文跟踪 + 元数据入库 + 按需全文抓取 + 语义检索 + 基础订阅”的核心系统；报告、PPT、配图和复杂可视化全部后置到二三期。**
